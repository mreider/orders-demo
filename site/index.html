orders-demo / SDv1 to SDv2 for OneAgent

A Spring Boot app plus a presentation on the Service Detection v1 to v2 migration for OneAgent-monitored workloads. SDv2 for OneAgent is available only on Latest Dynatrace tenants. Demos run against your own tenant.

View on GitHub Setup guide Presentation outline

The flow

Part 1: what is a service, really?

Entity-first to attribute-first. The workload as the unit. Three transport families. Where SDv1's dt.service.name fragmentation lives, and how SDv2 collapses it.

Part 2: naming and identity

Where service names come from, the metric-first query shift, and the service.name fix for Classic services you own today.

Part 3: endpoints, dependencies, and what's next

Every endpoint baselined, the http.route gap and the automatic fix that's coming, downstreams as tabs, and the pipeline-side naming plus SERVICE_DEPLOYMENT direction.

How it works

The presentation is sections, each with a short narrative, a slide deck, and a demo notebook that runs against your own Dynatrace tenant. A companion Spring Boot app fills in when the tenant doesn't already have a workload with the right shape.

ComponentWhat it is
app/Spring Boot monolith: two controllers, a Kafka consumer, a JDBC client
k8s/Manifests: two namespaces, Postgres + Redpanda per side, two app deployments in Classic + one in Latest, two loadgens
presentation/Presentation sections (markdown + demo notebook YAML + per-section Marp slides) plus one consolidated editable PPTX
scripts/up.shBrings up the whole stack on GKE in one command
scripts/load-demos.shLoads every demo notebook into your tenant with an environment share

Quick start

# After you've set the required env vars (see SETUP.md):
./scripts/up.sh
./scripts/load-demos.sh

Full prerequisites (GKE, Dynatrace tokens, dtctl) in the setup guide.

Releases

Each tagged release bundles the presentation as three artifacts on the releases page: the consolidated slide deck (PPTX), the demo notebooks (YAML), and the full presentation tree (tarball).
